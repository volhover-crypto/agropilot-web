# ТЗ — AgroPILOT / A PILOT (АгроЭлемент)
Техническое задание · ШАГ-1 · База: HANDOVER.md (2026-07-08)

## 1. Назначение
Мультиагентная офисная система цифровых фамильяров для A PILOT/АгроЭлемент
(Крым/Кубань), замещающая ERP/CRM/финучёт/планировщики/вики/контроль версий/
календари/коммуникации. Малая команда; люди = стратегия + верификация.

## 2. Архитектура (факт из кода)
- Объектно-ориентированный агро-B2B рабочий стол.
- Стек: Alpine.js (без сборки), Tailwind/Pico, ванильный JS, строковый
  innerHTML-рендер, hash-роутинг.
- Файлы:
  - index.html — auth-guard, login-модал, темы;
  - js/api.js — REST-клиент BFF :5555, /agropilot/api;
  - js/app.objects.js — ядро (~130 КБ): state + resolvers + вьюхи;
  - js/mock.objects.js — window.MOCKO демо-модель.
- Режимы: apiMode true(BFF)/false(mock); fallback на mock при пустом/ошибочном
  ответе; retry+refresh на 401/403.
- Персист: localStorage, ключ agropilot_data_v1 (persist/restore).

## 3. Доменная модель
Иерархия: Стратегия → Цели → Проекты/Сделки/Концепции → Задачи → Артефакты;
мягкая связь по industry + need_type.
Резолверы: clientById, clientName(c)=c.name||c.title||c.id, dealById;
агрегация objActivity() + vTimeline() + activityBlock().

## 4. AI-контур
- Один модератор ПЕТРУШКА (Level-2) + LLM-эндпоинты: orchChat, aiScore/enrich/
  followup/generate-kp/contract, aiDigest, aiRecommendations.
- Градации: AUTO / CONFIRM / HINT.
- Эвристика petReply() при !apiMode.
- ПЕТРУШКА — непрерывный агент, контекст = вся система; ежедневный анализ,
  рекомендации, ведение сценариев/индикаторов.
- v3.1 (13.07.2026): двухконтурный контекст и роль «монитор мультиопыта» — см. §9.

## 5. Функциональные требования
- 4 блока жизненного цикла клиента:
  - Блок1 Подготовка/Маркетинг/SMM/мониторинг источников;
  - Блок2 Работа с клиентом (воронка, договоры);
  - Блок3 Выполнение работ (реализация);
  - Блок4 Сервис и пост-договор.
- Над блоками — СТРАТЕГИЯ по сценарному планированию (VUCA): 2–4 сценария
  будущего + индикаторы + линии действий; гибкая; = системный промпт,
  относительно неё подбираются источники (агент/человек).
- Аудит действий: actor_name в истории сделок, вывод в таймлайн.

## 7. Известные баги / риски
- Patch B: actor_name в deal.history не реализован (главный пункт доработки).
- Опечатка 'Новый пект'.
- Массовый innerHTML — esc() неединообразно (XSS-риск на реальных данных).
- restore() (localStorage) до loadFromAPI() — приоритет источников не задан.
- STAGE_MAP из BFF (Проектирование/Выиграна) ≠ фильтры vMetrics
  (Реализация/Сервис) → конверсия неверна.
- Ветка master вместо main.

## 8. Ограничения
- IoT исключён полностью (нет датчиков/телеметрии).
- Правки только в исходники (без сборки).
- Прод-код меняется только после явного подтверждения заказчика.

## 9. Расширение v3.1 — монитор мультиопыта и верифицированные знания
Утверждено заказчиком 13.07.2026. Путь B сохраняется: новых агентов в UI нет,
МХ-роль — «шляпа» ПЕТРУШКИ; её навыки меряются общим порогом B→A.

- Двухконтурный контекст ПЕТРУШКИ: внутренний — вся система; внешний — только
  через реестр верифицированных источников и базы знаний (Qdrant/RAG); любые
  внешние данные — с цитированием источника.
- Реестр источников: жизненный цикл proposed → pending → verified → revoked;
  верификация одним пользователем (кворум = 1); trust/отзыв — только isManager().
  Стартовые коннекторы: arXiv, КиберЛенинка, открытые ресурсы. Среды мониторинга
  (профильные сайты, Telegram-каналы, соцсети) подключаются через SMM-раздел (Блок 1).
- Роль «монитор мультиопыта»: AUTO-сбор ux_signals (портал + внешние среды);
  проактивные микро-опросы пользователей ОБЯЗАТЕЛЬНЫ — каждый вопрос в лог
  agent_questions (asked|deferred|answered|expired), ответить можно позже из лога;
  прогнозы рынка — HINT с привязкой к индикаторам Стратегии; рекомендации типа
  system_improvement; весь цикл — в аудит с actor_name.
- Новые сущности: sources[], knowledge_bases[], ux_signals[], insights[],
  agent_questions[]; recommendation.kind += system_improvement | market_forecast.
- Реализация — Этап-2 (M10 Источники, M11 Базы знаний/RAG, M12 МХ-мониторинг);
  контракты — CONTRACTS.md §8–§10; флаги SOURCES_READY/KNOWLEDGE_READY/UX_READY.
