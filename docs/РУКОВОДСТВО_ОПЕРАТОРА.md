# Руководство оператора AgroPILOT — как вести работу, если ты не программист

> Эта памятка объясняет простыми словами, как довести до конца задачи Этапа-2 (M10),
> работая в паре с ассистентом Perplexity. Тебе НЕ нужно уметь программировать.
> Твоя роль — «руки и глаза»: запускать команды на сервере и вставлять их вывод обратно ассистенту.
> Ассистент — «голова»: он думает, пишет код и правит файлы в репозитории.

---

## 1. Кто что делает (роли)

| Кто | Что делает |
|---|---|
| **Ты (оператор)** | Заходишь на сервер по SSH, копируешь команду от ассистента, вставляешь, копируешь ответ терминала обратно в чат. Нажимаешь «коммить», когда просят. |
| **Ассистент Perplexity в Comet** | Думает, пишет код, правит файлы прямо в репозитории (GitHub), даёт тебе команды. |
| **Сервер (root@mdked)** | Где живёт приложение. Ты им управляешь через терминал. |
| **Репозиторий (GitHub)** | Хранилище кода. Ассистент туда пишет, сервер оттуда забирает командой `git pull`. |

Главное правило: **ассистент ничего не знает о текущем состоянии, пока ты не покажешь ему вывод команды.** Он не видит сервер. Поэтому вы всё время обмениваетесь: он команду → ты вывод.

## 2. Что тебе нужно открыть перед работой

1. **Терминал на своём ПК** и подключение к серверу:
   ```
   ssh root@176.12.74.69
   ```
   Признак, что ты «внутри» сервера: строка вида `root@mdked:~#`.

2. **Браузер Comet** с ассистентом Perplexity в пространстве (Space) «AgroPILOT».

Держи оба окна рядом — ты будешь копировать между ними.

## 3. Как выглядит один рабочий цикл (простыми словами)

Представь диалог. Ассистент всегда идёт маленькими шагами и после каждого ОСТАНАВЛИВАЕТСЯ и ждёт тебя.

```
Ассистент: "Шаг 1. Вставь, пожалуйста, эту команду и пришли мне что она вывела:
            <команда>
            ОЖИДАЮ УВИДЕТЬ: <что должно появиться>"

Ты:        [копируешь команду в терминал сервера, жмёшь Enter,
            выделяешь весь ответ терминала, вставляешь в чат]

Ассистент: "Вижу. PASS (всё как ожидалось). Шаг 2. ..."
```

Ты почти никогда не набираешь команды сам — ты их **копируешь** у ассистента. Твоя задача — аккуратно перенести туда-обратно.

---

## 4. Три кнопки, которые ты будешь нажимать чаще всего

### Копировать команду → в терминал
Ассистент даёт команду в сером блоке. Наведи, нажми «копировать», вставь в терминал сервера (Ctrl+Shift+V в большинстве терминалов), Enter.

### Копировать ответ терминала → в чат
Выдели всё, что терминал написал в ответ (мышкой или Ctrl+A в области вывода), скопируй, вставь в чат ассистенту. **Копируй целиком, даже если выглядит непонятно** — ассистенту важна каждая строка, особенно слова `error`, `404`, `FAIL`.

### «Коммить» / «ок, поехали»
Когда ассистент показал, что он изменил (это называется diff), и написал «жду слово коммить» — если тебе всё ок, просто напиши в чат: `коммить`. Это разрешение сохранить изменения. Аналогично `ок, поехали` — разрешение начать.

---

## 5. Слова ассистента и что они значат

| Слово | Перевод на человеческий |
|---|---|
| **PASS** | Шаг прошёл успешно, идём дальше |
| **FAIL** | Что-то не так, ассистент будет разбираться — не пугайся |
| **diff** | Показ «что именно изменится в коде» перед сохранением |
| **коммит** | Сохранить изменения в репозиторий (как «сохранить документ») |
| **push** | Отправить сохранённое в GitHub (в облако) |
| **git pull** | Забрать на сервер свежую версию из GitHub |
| **restart** | Перезапустить приложение, чтобы оно увидело новый код |
| **ПРЕДПОЛОЖЕНИЕ** | Ассистент честно говорит «я не уверен, надо проверить» — это хорошо, значит он не выдумывает |
| **agent_questions** | Ассистенту не хватило данных, он задаёт вопрос вместо догадки |

## 6. Твои простые правила безопасности

1. **Никогда не жми «коммить», если не понял, что меняется.** Спроси: «объясни простыми словами, что этот шаг делает». Ассистент обязан объяснить.
2. **Опасные команды** (со словами `rm`, `drop`, `restart`, `--force`) ассистент ОБЯЗАН предупредить заранее. Если команда со `rm` пришла без предупреждения — остановись и спроси «это точно безопасно?».
3. **Вставляй вывод целиком.** Не сокращай, не убирай «страшные» строки — именно они и нужны.
4. **Один шаг за раз.** Если ассистент прислал сразу 5 команд без остановок — попроси: «давай по одному шагу, я не программист».
5. **Не чини сам.** Если что-то сломалось — просто вставь ошибку ассистенту и напиши «сломалось, вот вывод». Он разберётся.

## 7. Как понять, что задача (Issue) выполнена

В конце каждой задачи ассистент показывает **чек-лист приёмки**. Все пункты должны быть отмечены с доказательством (выводом команды):

```
[✓] контракт подтверждён
[✓] таблица в базе создана
[✓] проверка синтаксиса без ошибок
[✓] diff одобрен тобой
[✓] изменения отправлены в GitHub (проверено)
[✓] приложение перезапущено
[✓] эндпоинт отвечает 200 (не 404)
[✓] раздел на сайте показывает данные
[✓] запись в HANDOVER добавлена
```

Когда все галочки стоят — можно закрыть Issue на GitHub (кнопка «Close issue») и переходить к следующей.

---

## 8. Если ты запутался — три спасательные фразы

- **«Объясни проще, я не программист»** — ассистент переформулирует.
- **«Давай по одному шагу»** — если завалил кучей команд.
- **«Покажи, где мы сейчас и что дальше»** — вернёт тебя к карте прогресса.

## 9. Порядок работы над Этапом-2

1. Открой терминал + Comet.
2. В новом чате Space вставь **стартовый промпт** (он в этом же репо / у ассистента).
3. Ассистент попросит синхронизацию — выполни команды, вставь вывод.
4. Он предложит начать с задачи **#2 Sources** (самая важная). Согласись: `ок, поехали`.
5. Идите по шагам. Ты копируешь туда-обратно и нажимаешь «коммить», когда согласен.
6. В конце — чек-лист, закрываешь Issue, следующая задача.

Не спеши. Правильно — это медленно и по одному шагу, с проверкой на каждом.
Ассистент отвечает за код и безопасность; ты — за «глаза» (показать реальный вывод) и «стоп-кран» (не жать коммит вслепую).
