# ТЗ Этап-2 — AgroPILOT (Stage 2)

> Монолит. Порядок: §0 → §1–§6 → §7 → §8 → §9 → Приложение A.
> Рабочий протокол: docs/MAX_AS_COMPUTER.md · шаблон треда: docs/PROMPT_STEP_TEMPLATE.md · контракты: CONTRACTS.md · журнал: HANDOVER.md

## 0. Правила работы (строго обязательные)

**Режим исполнения:** работа ведётся в Perplexity Space «AgroPILOT» (MAX-AS-COMPUTER),
БЕЗ браузерного режима «Компьютер»/Comet. Роли:
- **Архитектор (заказчик/ревьюер):** diff-ревью, подтверждение «ОК», верификация по GitHub API (raw по full SHA).
- **AI-агент (исполнитель):** думает, пишет код/диффы, вносит изменения в репозиторий; НЕ действует автономно между шагами.
- **Оператор (руки/глаза):** применяет на сервере через SSH, выполняет команды и вставляет вывод ЦЕЛИКОМ.

**Источник правды — репозиторий и прод, а не память и не файлы, загруженные в Space**
(HANDOVER/ROADMAP в Space могут устаревать). Непроверённый командой факт = ПРЕДПОЛОЖЕНИЕ.

1. **Repository-only.** Все изменения — только через репозиторий; малые точечные диффы (small diffs).
2. **Diff-ревью.** Перед каждым `Commit & Push` агент показывает DIFF всех изменённых файлов и запрашивает явное «ОК».
3. **Верификация.** После пуша — чтение raw-контента по новому full SHA (GitHub API), построчная сверка.
4. **Рестарт.** Для backend-изменений — `restart agropilot-backend.service` (НЕ agropilot.service).
5. **Контракты.** Фиксируются в `CONTRACTS.md` ДО реализации фронта (contract-first).
6. **Пошаговость + СТОП.** После каждого шага — остановка и отчёт: что сделано, как проверить, что дальше.
7. **Без Issues.** GitHub Issues/Milestone для трекинга НЕ используем. Трекинг — по шагам этого ТЗ + записи в `HANDOVER.md`.
8. **Правило остановки.** Если фактическая структура файлов/данных отличается от ожидаемой — остановиться и доложить, не домысливать.
9. **Токен-экономия (обязательно):**
   - ТЗ хранится в репо (`docs/TZ_STAGE2.md`); в чат подаётся ТОЛЬКО текущий шаг, а не весь документ.
   - Каждый шаг выполняется в ОТДЕЛЬНОМ треде (fresh thread); состояние между шагами живёт в `HANDOVER.md`.
   - Каждый тред начинается с «СИНХРОНИЗАЦИИ КОНТЕКСТА» (см. §7), а не с перечитывания всего кода.
   - Мультимодельное ревью кода — в отдельном треде промптом «найди расхождения с контрактом и потенциальные 500».

## 1. Decision Log (зафиксировано на старте)

1. **GitHub Issues — не используем.** Трекинг: ТЗ (docs/TZ_STAGE2.md) + пошаговый workflow (DIFF → ОК → Commit → SHA-verify) + записи в HANDOVER.md.
2. **Очередь `proposed` — на дашборде «Мой день»** отдельным виджетом/полем (не в общем разделе «Мониторинг»).
3. **Маршрутизация `proposed`:**
   - 3.1. Сначала → автору исходного задания на мониторинг (`added_by`).
   - 3.2. Если автор недоступен/не определён → по компетенции (пользователям с совпадающим `competencies[]`).
   - 3.3. Fallback «некуда доставить» → тоже по компетенции.
4. **Право Approve/Reject `proposed`** — только у адресата (receiver), которому предложение доставлено по правилам п.3 (не «вся роль»).

## 2. Цель

Довести до полноценной работы 7 ключевых разделов, внедрить «Справочник» сущностей и «Справочник пользователей» (компетенции/права), реализовать сквозные стратегические задачи и привести ПЕТРУШКУ к целевой модели: системный агент, который держит в контексте всю внутреннюю информацию системы + управляемые источники мониторинга, мониторит/оценивает/рекомендует через призму стратегии и адресно приносит наводки (`proposed`) на дашборд «Мой день» нужному человеку.

## 3. Scope (блоки работ)

- **Блок A** — первая волна из 7 разделов: Стратегия, Цели, Мониторинг, Контекст, Соцсети, Входящие клиенты, Сделки.
- **Блок B** — «Справочник» сущностей (read-only дерево-архив, как проводник Windows).
- **Блок C** — стратегические задачи (сквозной контекст в `owlContext()`).
- **Блок D** — ПЕТРУШКА + реестр источников `/v1/sources` + `proposed` на «Мой день» с адресной маршрутизацией.
- **Блок E** — «Справочник пользователей»: команда, компетенции, права (основа маршрутизации и RBAC).

## 4. Non-goals (в этом этапе НЕ делаем)

- RAG / векторизацию «Справочника» — отложено в v2 (Приложение A).
- Реальные внешние скрейперы/интеграции каналов — допускается mock-слой генерации сигналов.
- Авто-рекомендации по cron/event-триггерам в проде — v2.
- GitHub Issues для трекинга.
- Расширенную доменную модель сущностей (полный набор полей) — в Приложение A.

## 5. Функциональные требования

### 5.1. Блок E — Справочник пользователей / компетенций / прав

Опирается на существующий задел M9 «Навыки команды»; люди/роли размещаются в группе «Объекты».

**Модель `user`:**

| Поле | Назначение |
|---|---|
| `id` | Идентификатор |
| `name` | Имя/название |
| `role` | Роль (`admin`, `manager`, `smm`, `engineer`, … — см. §8.1) |
| `competencies[]` | Зоны ответственности/ниши (напр. «агро-инфополе», «поставщики оборудования», «сопредельные рынки») |
| `permissions[]` | Что может (создавать стратег.задачи, добавлять источники типа X, одобрять `proposed`) |
| `assigned_tasks[]` | Задания на мониторинг, которые пользователь создал / за которые отвечает |
| `status` | `active` / `inactive` |

**Назначение блока:** (а) валидация действий (RBAC), (б) адресная маршрутизация `proposed`.

**DoD Блока E:**
1. Есть реестр пользователей с ролью, компетенциями, правами, статусом.
2. Задание на мониторинг хранит автора (`added_by`).
3. Права реально ограничивают действия (создание задач, добавление источников, Approve/Reject `proposed`).

### 5.2. Блок C — Стратегические задачи (сквозной контекст)

**Модель `strategy_task`:** `id`, `title`, `description`, `priority`, `status`, `monitoring_focus[]` (ключевые слова / ниши / рынки), `owner`, (опц.) `linked_scenario`.

**Поведение:**
- Активные задачи обязаны попадать в `owlContext()`.
- MIA / ПЕТРУШКА используют `monitoring_focus` как призму фильтрации инфополя и оценки сделок.
- Создание задачи: роль `manager` / `admin` (`isManager()`).

**DoD Блока C:**
1. Задача создаётся в UI «Стратегия» → появляется в `owlContext()`.
2. Лента мониторинга помечает/фильтрует наблюдения по `monitoring_focus`.
3. При оценке сделки ПЕТРУШКА ссылается на релевантную стратегическую задачу.

### 5.3. Блок D — ПЕТРУШКА + источники + `proposed` на «Мой день»

- **D-1. Аудит (первым, до изменений):** проверить фактический состав `owlContext()`, сформировать gap-отчёт «как есть → как должно». Правило остановки — §0.8.
- **D-2. Целевая модель:** `owlContext()` = контекст route/объекта + стратегия + активные стратегические задачи + активные источники мониторинга.
- **D-3. Реестр источников `/v1/sources`:** `id`, `type` (`news`/`supplier`/`competitor`/`market`/`tech`), `url`/`endpoint`, `keywords[]`, `linked_strategy_task`, `added_by`, `status` (`active`/`proposed`/`disabled`/`rejected`).
- **D-4. Кто добавляет источники:** SMM → `news`/контент/соцсети; инженер/член команды → `supplier`/`tech`/`market`; ПЕТРУШКА → только со статусом `proposed`.
- **D-5. Маршрутизация `proposed`:** (1) автору задания (`added_by`) → в его «Мой день»; (2) если автор неактивен/не определён → по компетенции (совпадение `competencies[]`); (3) fallback «некуда доставить» → тоже по компетенции; (4) Approve/Reject — только адресат. Служебные поля: `receiver_user_id` (кому доставлено сейчас), `routing_reason` (`added_by` | `competency`).
- **D-6. UI «Мой день»:** виджет «Предложения на мониторинг (N)» с действиями Одобрить / Отклонить. Одобрено → `status = active` и попадает в `owlContext()`; отклонено → `status = rejected`.

**DoD Блока D:**
1. `owlContext()` включает route/объект + стратегию + активные стратег.задачи + активные источники.
2. `/v1/sources` поддерживает типы `news/supplier/competitor/market/tech` и роль-автора.
3. ПЕТРУШКА может создать источник со статусом `proposed`.
4. `proposed` попадает в «Мой день» правильного пользователя по D-5; Approve/Reject доступны только адресату.
5. В ленте мониторинга видно, по какому источнику и какой стратегической задаче получено наблюдение.

### 5.4. Блок A — 7 разделов (должны работать «полноценно»)

| # | Раздел | Требование | Эндпоинт |
|---|---|---|---|
| A-1 | Стратегия | CRUD стратегии + стратег.задачи (Блок C) | `/v1/strategy`, `/v1/strategy/tasks` |
| A-2 | Цели | CRUD целей + связи со стратегией/задачами | `/v1/goals` |
| A-3 | Мониторинг | Лента наблюдений по активным источникам + фильтр/пометка по стратегии | `/v1/monitoring` (+ `/v1/sources`) |
| A-4 | Контекст | Просмотр агрегированного контекста ПЕТРУШКИ | `/v1/context` |
| A-5 | Соцсети | Каналы / архив постов SMM | `/v1/content` |
| A-6 | Входящие клиенты | Первичный источник клиентов (не дерив из `deals`) | `/v1/clients` |
| A-7 | Сделки | Реальные сделки + оценка/комментарии ПЕТРУШКИ | `/v1/deals` |

**DoD Блока A:**
1. Каждый раздел — отдельный роутер + модель по паттерну `deals`, зарегистрирован в `main.py`.
2. Клиенты берутся из `/v1/clients`, а не деривятся во фронте через `loadClients()`.
3. Фронт грузит данные с бэкенда; в консоли браузера 0 ошибок JS.
4. Флаги `*_READY` проставлены в `CONTRACTS.md`; сервис перезапущен.

### 5.5. Блок B — «Справочник» сущностей (Tree-view архив)

**Назначение:** единый древовидный архив всех сущностей системы (как проводник Windows) — артефакты (КП/договоры/схемы), контент (посты — архив генерации SMM), сделки, клиенты, цели, стратегические задачи, источники, пользователи.

**Принцип:** read-only агрегатор (read-model) поверх существующих таблиц/реестров — без дублирования данных.

**Модель листа:** `id`, `type`, `title`, `created_at`, `source_agent` (MIA / SMM / SPA / ПЕТРУШКА / человек), deep-link на оригинальную карточку.

**API:**
- `GET /v1/catalog/tree?node=<id>` — дерево с lazy-load дочерних узлов.
- `GET /v1/catalog/search?q=` — поиск (питает сквозную палитру `Ctrl+K`).

**UI («Проводник»):** дерево слева (expand/collapse, иконки по типу), превью листа справа + «Открыть карточку» (deep-link); переиспользовать палитру `Ctrl+K`.

**Ветки:** Артефакты / Контент / Сделки / Клиенты / Цели / Стратег.задачи / Источники (по типам) / Пользователи (по ролям → карточка).

**DoD Блока B:**
1. `/v1/catalog/tree` возвращает корректную иерархию; lazy-load работает.
2. Артефакты (КП/договоры) и посты (архив генерации) видны как листья в своих ветках.
3. Клик по листу открывает оригинальную карточку.
4. Дерево — read-only отражение реальных данных, без рассинхрона с источниками.

## 6. Связи между сущностями (целевая логика)

```
Пользователь(competencies/permissions) → маршрутизация proposed + RBAC
Стратегическая задача → цели, мониторинг, контекст, оценка клиентов/сделок, рекомендации ПЕТРУШКИ
Источник → сигналы, привязка к стратег.задаче и added_by, попадание в контекст
Сигнал → может стать клиентом / сделкой / задачей / темой поста / рекомендацией
Сделка → клиент + цели + стратегия + артефакты (КП/договор), оценивается ПЕТРУШКОЙ
Артефакт → сделка/клиент/стратегия, отображается листом в «Справочнике»
```

Общий поток: `Стратегия → Контекст → Мониторинг → Сигналы → Клиенты/Сделки/Контент → Рекомендации → Новые действия`.

## 7. Порядок выполнения (строго шаг за шагом)

**Механика:** каждый шаг = отдельный тред в Space. Тред открывается «Синхронизацией контекста»,
далее задание на один блок, затем цикл `контракт → diff → «ОК» → commit → verify SHA → restart → запись в HANDOVER`.

| Шаг | Блок | Содержание | Выход |
|---|---|---|---|
| **1** | D-1 | Аудит `owlContext()` → gap-отчёт «как есть → как должно» | отчёт → ОК |
| **2** | E | Справочник пользователей / компетенций / прав (основа маршрутизации и RBAC) | ОК |
| **3** | C | Стратег.задачи (`/v1/strategy/tasks`) + инжект в `owlContext()` | ОК |
| **4** | D | Источники (`/v1/sources`) + `proposed` + виджет «Мой день» (маршрутизация D-5) | ОК |
| **5** | A | Разделы по очереди: clients → deals → content → monitoring → goals/context/strategy; каждый: restart + флаг `*_READY` | ОК по шагам |
| **6** | B | «Справочник» сущностей (read-model поверх готовых реестров) | финальное ОК |

**Цикл одного шага:**
1. Новый тред → вставить «Стартовый промпт на шаг» (см. `docs/PROMPT_STEP_TEMPLATE.md`).
2. Оператор выполняет блок синхронизации и вставляет вывод ЦЕЛИКОМ.
3. Агент фиксирует состояние по факту → предлагает контракт в `CONTRACTS.md` → СТОП, ждёт «ок, поехали».
4. Роутер + модель по паттерну `backend/deals` → регистрация в `backend/main.py`.
5. DIFF → «ОК» → Commit & Push → verify raw по full SHA (grep-маркер).
6. `restart agropilot-backend.service` → `curl 127.0.0.1:5555/agropilot/api/v1/<ep>` = `{ok,data}`.
7. Запись в `HANDOVER.md` (шаг, SHA, что закрыто) → закрыть тред.

## 8. Открытые вопросы (уточнить при реализации)

1. Набор ролей: базовый (`admin`, `manager`, `smm`, `engineer`) или нужны доп. роли? По умолчанию агент берёт базовый и предлагает расширение по факту.
2. «Справочник»: строго read-only или в перспективе нужен drag-n-drop / переорганизация дерева?
3. Дашборд «Мой день»: новый раздел или расширение существующего экрана; какие ещё блоки кроме `proposed` (задачи, сделки, наблюдения)?

## 9. Риски и меры

| Риск | Мера |
|---|---|
| Рассинхрон «Справочника» с источниками | Только read-model (Блок B) |
| Перегруз MVP полями | Расширенные модели → Приложение A |
| ПЕТРУШКА работает «не как задумано» | Сначала аудит (Шаг 1), фиксация gap-отчёта, потом изменения |
| «Мусорные» источники | Только `proposed` + ручное подтверждение адресатом |
| Потеря адресности `proposed` | Маршрутизация зависит от Блока E → он идёт до Блока D |

## Приложение A. Backlog расширения (v2, не для MVP)

- Расширенная карточка `strategy_task` (связи с клиентами/сделками/сигналами, география, критерии важности, история влияния).
- Сущность «сигнал мониторинга» с типизацией + преобразования сигнал → клиент / сделка / задача / пост.
- Расширенная оценка клиентов/сделок (маржинальность, вероятность закрытия, риски, next-best-action).
- Полноценный RBAC (`permissions[]`) во всех разделах.
- RAG-архитектура: мета-промпт из стратегии + векторизация «Справочника»; фоновая аналитика (cron / event-driven).
- Возможный отдельный «Инбокс» (единая приёмная «требует решения»), если появятся типы `proposed` кроме источников.
