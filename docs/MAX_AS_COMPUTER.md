# MAX-AS-COMPUTER — стратегия реализации M10 силами Perplexity Max

> Как вести обычный чат Perplexity Max так, чтобы он работал максимально близко к режиму Computer.
> Источник правды — repo + прод (не память модели). "Руки разработчика" — оператор (пользователь).
> Читать в начале каждой Max-сессии по проекту AgroPILOT. Связка: `ROADMAP_M10.md`, `HANDOVER.md` §15.

## 1. Gap: Computer vs Max

| Возможность Computer | В Max | Замена в Max |
|---|---|---|
| Sandbox (пишет/запускает код) | нет | Модель выдаёт код блоком → оператор вставляет в файл |
| Push в GitHub | нет | Модель даёт `git`-команды verbatim → оператор выполняет |
| SSH на сервер | нет | Модель даёт shell-команды → оператор вводит в терминал |
| curl-верификация | нет | Модель даёт curl → оператор вставляет вывод обратно |
| Чтение repo/прода | нет | Оператор копирует вывод; модель работает только по нему |
| Subagent-параллелизм | нет | Один тред = одна M10-задача; ревью вторым проходом/моделью |
| Память между сессиями | Spaces + этот файл | HANDOVER.md + ROADMAP_M10.md = внешняя память |

Ключ: в Computer правда = sandbox; в Max правда = оператор (repo+сервер). Непроверенный факт = ПРЕДПОЛОЖЕНИЕ.

## 2. Операционный протокол (system-instructions для Space/треда)

```
Ты работаешь в режиме MAX-AS-COMPUTER для проекта AgroPILOT.
У тебя НЕТ доступа к файлам, GitHub, серверу. Твои "руки" — оператор,
который выполняет команды и вставляет вывод. Repo и прод = единственный источник правды.

ЖЕЛЕЗНЫЕ ПРАВИЛА:
1. НИКОГДА не выдумывай пути, имена эндпоинтов, содержимое кода, статусы.
   Факт без подтверждения выводом команды в этом треде = ПРЕДПОЛОЖЕНИЕ; запроси проверку.
2. Работай ШАГАМИ: один шаг = одно действие + одна проверка. После шага — СТОП, жди вывод.
3. Каждая команда — verbatim в блоке + строка "ОЖИДАЮ УВИДЕТЬ: ...". После вывода — PASS/FAIL.
4. Перед коммитом покажи полный diff и жди слово "коммить".
5. Перед необратимым (push, rm, restart, drop) — предупреди и жди подтверждения.
6. Код давай ЦЕЛИКОМ для вставки в файл (с якорем, если правка частичная).
7. Backend-роутер — строго по паттерну backend/deals: AsyncSession, Depends(get_db,get_current_user),
   конверт {ok,data}, NotFoundError, регистрация в main.py.
8. После push — проверка raw.githubusercontent.com/.../main/<файл>.
9. После backend-изменений — restart agropilot-backend.service (НЕ agropilot.service — masked).
10. В конце задачи — чек-лист приёмки, все пункты PASS с доказательством (вывод команды).
11. Сомневаешься — задай вопрос, не угадывай.
12. Русский язык, без воды, формат "команда → ожидание → факт → PASS/FAIL".
```

## 3. Антигаллюцинационные барьеры

1. **Contract-first** — до кода фиксируем контракт эндпоинта (метод, путь, поля). Код обязан ему соответствовать.
2. **Проверка после push** — `curl -s raw.githubusercontent.com/volhover-crypto/agropilot-web/main/<файл> | grep <маркер>`. Нет маркера → push не долетел.
3. **Синтаксис до коммита** — `python3 -m py_compile <f>` / `node --check <f>`. FAIL → не коммитим.
4. **Runtime-verify** — `curl -s http://127.0.0.1:5555/agropilot/api/v1/<ep> | head -c 200` = `{"ok":true,"data":...}`. 404/500 → не закрыто.
5. **Diff-gate** — `git diff` перед коммитом; оператор глазами сверяет.
6. **Цитирование факта** — любое утверждение о состоянии сопровождается командой-пруфом. Нет пруфа = предположение.

## 4. Эталонный цикл (пример M10-2 Sources) — тиражируется на все задачи

```
Шаг 0. Оператор вставляет backend/deals/{routes,models}.py (эталон) + ls backend/
Шаг 1. Модель фиксирует КОНТРАКТ:
        GET   /api/v1/sources        -> {ok, data:[{id,type,url,keywords[],active,created_at}]}
        POST  /api/v1/sources        -> {ok, data:{...}}
        PATCH/DELETE /sources/{id}
        [СТОП — подтвердить контракт]
Шаг 2. Миграция таблицы sources → psql → \d sources. ОЖИДАЮ колонки. PASS/FAIL
Шаг 3. backend/sources/{models,routes,__init__}.py ЦЕЛИКОМ → py_compile. PASS/FAIL
Шаг 4. Правка main.py (регистрация) → git diff → [СТОП — "коммить"]
Шаг 5. git add+commit+push → curl raw...main.py | grep sources_router. PASS/FAIL
Шаг 6. systemctl restart agropilot-backend.service
Шаг 7. curl 127.0.0.1:5555/.../sources → {"ok":true,"data":[]}. PASS/FAIL
Шаг 8. Фронт (loadSources уже есть) → hard-reload → раздел рендерит.
Шаг 9. Чек-лист приёмки → HANDOVER.
```

## 5. Шаблоны

### 5.1 Чек-лист приёмки (на каждую M10-задачу)
```
[ ] Контракт зафиксирован и подтверждён
[ ] Миграция применена (\d <table> = нужные колонки)
[ ] py_compile / node --check без ошибок
[ ] git diff показан и одобрен
[ ] push долетел (raw...grep <маркер>)
[ ] restart agropilot-backend.service
[ ] curl эндпоинта = 200 + {ok,data}
[ ] раздел UI рендерит (Ctrl+Shift+R)
[ ] запись в HANDOVER добавлена и запушена
```

### 5.2 Формат шага (модель обязана держать)
```
### Шаг N: <название>
Команда:
    <verbatim>
ОЖИДАЮ УВИДЕТЬ: <признак успеха>
[вставьте вывод — жду перед следующим шагом]
```

### 5.3 Кросс-задача «обязательное цитирование» (M10-7/8)
Каждый AI-ответ оркестратора возвращает `sources[]` с URL/id записи-основания; UI показывает.
Ответ без источника = невалиден. Антигаллюцинация на уровне продукта.

### 5.4 Кросс-задача `agent_questions`
Недостаточно данных → агент пишет строку в `agent_questions` и НЕ выдумывает; фронт показывает оператору.
Продуктовый аналог правила №11 протокола.

## 6. Мультимодельный контроль (дешёвый model-council в Max)

| Проход | Модель | Задача |
|---|---|---|
| План/контракт | reasoning-модель | архитектура эндпоинта |
| Генерация кода | сильная code-модель | роутер/миграция |
| Независимое ревью | ДРУГАЯ модель, новый тред | "найди расхождения с контрактом и потенциальные 500" |
| Финальный verify | человек + curl | факт на проде |

Правило: код модели A ревьюит модель B в отдельном треде без контекста автора. Расхождение → назад к A.

## 7. Durable-правила (из HANDOVER §15)
- Роутеры — по паттерну backend/deals; регистрация в main.py под /agropilot/api/v1; активация = restart agropilot-backend.service.
- Рабочий сервис — agropilot-backend.service (venv); agropilot.service — masked.
- После правок — py_compile/node --check, push, raw-проверка, runtime-QA.
- Полное восстановление файла может затирать точечные фиксы — сверять после restore.
- Статус Issue проверять в GitHub, не в markdown.
